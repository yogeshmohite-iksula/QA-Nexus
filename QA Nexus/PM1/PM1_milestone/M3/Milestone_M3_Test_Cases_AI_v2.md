## 1. Executive Summary

M3 ships the **first true AI-native QA workflow**: **Composer** ⓘ (Test Case Generator) for PRD → test cases, and **Curator** ⓘ (Duplicate Detection) for semantic dedup of generated cases against existing ones. Built on the M2 KB foundation: when QA generates test cases for a requirement, Composer fetches relevant KB chunks via semantic search, injects them as grounded context, and asks Groq's `openai/gpt-oss-120b` to emit **JSON-schema-validated test cases** in either traditional step-based or BDD Gherkin format.

**Core value delivered:**

- QA Engineer drafts a requirement (or imports from Jira). One click → Composer ⓘ produces 8-15 test cases in 30 seconds, grounded in project KB. Each case has confidence score, rationale, and source-chunk citations.
- Near-duplicate cases (semantic similarity > 0.85) auto-flagged by Curator ⓘ before save. Reviewer chooses dedupe / keep / merge.
- Test case → requirement linking (RTM) auto-populated when Composer runs from a requirement.
- Bulk import (F16c) for legacy CSV/Excel test case migration.

**Frames going live:** F14 (Requirements), F14m1/m2/m3 (Edit/Link/Convert modals), F16a (Test Case Method Chooser), F16b (A1 Generate from Requirement), F16c (Bulk Import).

**Tests target:** ~80 net new (40 BE unit + 25 e2e + 15 FE).

**Exit gate:** A QA Engineer can sign in, open a requirement, click "Generate test cases", review 10 cases, dedupe 1, edit 2, save 9 — with full audit chain integrity and PII guards verified.

### EP-029 — Composer ⓘ Test Case Generator (internal code: A1)

```
POST /api/projects/:projectId/test-cases/generate
```

**Body:**

```typescript
{
  source: 'requirement' | 'description' | 'jira',
  requirementId?: string,     // when source='requirement'
  description?: string,        // when source='description'
  jiraIssueId?: string,        // when source='jira'
  format: 'step' | 'gherkin',
  count: number,               // 1..20
  promptHints?: string,        // optional: "focus on edge cases", etc.
  kbContext?: {
    enabled: boolean,
    topK?: number,             // default 5 chunks
  },
}
```

**Response:**

```typescript
{
  generationRunId: string,
  cases: Array<{
    title: string,
    description: string,
    format: 'step' | 'gherkin',
    steps?: Array<{step: string, expected: string, data?: object}>,
    gherkin?: string,
    preconditions: string[],
    priority: 'P0' | 'P1' | 'P2' | 'P3',
    confidenceScore: number,    // 0.00 - 1.00
    rationale: string,
    sourceChunkIds: string[],   // KB chunk UUIDs cited
  }>,
  metadata: {
    llmProvider: 'groq',
    llmModel: 'openai/gpt-oss-120b',
    inputTokens: number,
    outputTokens: number,
    durationMs: number,
    chunksRetrieved: number,
  }
}
```

**Pipeline:**

1. Validate input (Zod) + RBAC + project ownership
2. **Build context:**
   - If `source='requirement'`: fetch requirement + linked acceptance criteria
   - If `kbContext.enabled`: embed input → pgvector top-K chunks via M2 chunk-search
3. **Construct prompt** (see §6.1 prompt template)
4. **Call Groq** with `response_format: { type: 'json_schema', json_schema: TEST_CASE_BATCH_SCHEMA }`
5. **Validate response** against Zod schema. If invalid: retry once with stricter system prompt. If 2x invalid → fallback to llama-4-scout. If still invalid → return error.
6. **A2 dedup pre-check** (optional, if `?withDedup=true`): embed each generated case title+steps → check against existing project cases via pgvector. Mark `dedupeWarning: true` for similarity > 0.85.
7. **Insert TB-022 generation_run row** (audit) + return cases (NOT yet saved to test_case table — UI confirms before bulk insert)
8. Audit log: `test_case_generated` event with PII-redacted payload (no requirement description, no chunk text — only IDs + counts)

**Token budget:** ~10K input + 4K output per generation = ~50 generations per Groq RPD. With 8 users × 3 generations/day = 24 calls. Headroom = 4×.

## 6. AI Pipeline Details

### 6.1 Composer ⓘ Prompt Template (canonical — DO NOT improvise; lock in ADR-013)

**System prompt:**

```
You are Composer (internal code A1), the Test Case Generator agent
for QA Nexus. Your job is to generate high-quality, executable test
cases from a software requirement and (optionally) project knowledge-
base context.

Rules:
1. Generate EXACTLY {count} test cases. Do NOT exceed.
2. Every case must have a confidence score 0.00-1.00. Score < 0.6
   means human review required.
3. Cite source KB chunks (sourceChunkIds) when context informs the
   case. Empty array if generated from requirement alone.
4. Format: {format} ('step' = step-based, 'gherkin' = BDD Given/When/
   Then).
5. Cover positive flows, negative flows, edge cases, and at least
   one boundary condition.
6. NEVER invent product features not present in the input. If the
   requirement is unclear, set confidenceScore < 0.5 and explain
   in rationale.
7. Output MUST be valid JSON conforming to the provided schema.
```

**User prompt (template):**

```
REQUIREMENT
-----------
{requirement_title}

{requirement_description}

ACCEPTANCE CRITERIA
-------------------
{acceptance_criteria_list}

KB CONTEXT (top {topK} chunks from project knowledge base)
----------------------------------------------------------
{chunk_id}: {chunk_text}
{chunk_id}: {chunk_text}
...

USER HINTS
----------
{prompt_hints OR "none"}

Generate {count} test cases in {format} format.
```

**JSON schema** (passed to Groq via `response_format`):

```json
{
  "type": "object",
  "properties": {
    "cases": {
      "type": "array",
      "minItems": 1,
      "maxItems": 20,
      "items": {
        "type": "object",
        "properties": {
          "title": {"type": "string", "maxLength": 200},
          "description": {"type": "string", "maxLength": 1000},
          "format": {"enum": ["step", "gherkin"]},
          "steps": {
            "type": "array",
            "items": {
              "type": "object",
              "properties": {
                "step": {"type": "string"},
                "expected": {"type": "string"},
                "data": {"type": "object"}
              },
              "required": ["step", "expected"]
            }
          },
          "gherkin": {"type": "string"},
          "preconditions": {"type": "array", "items": {"type": "string"}},
          "priority": {"enum": ["P0", "P1", "P2", "P3"]},
          "confidenceScore": {"type": "number", "minimum": 0, "maximum": 1},
          "rationale": {"type": "string"},
          "sourceChunkIds": {"type": "array", "items": {"type": "string", "format": "uuid"}}
        },
        "required": ["title", "description", "format", "preconditions", "priority", "confidenceScore", "rationale", "sourceChunkIds"]
      }
    }
  },
  "required": ["cases"]
}
```

### 6.2 Curator ⓘ Dedup Logic (internal code A2)

**On-the-fly:** for each candidate, embed `title + first 200 chars of description + first step` → 1024-dim vector → pgvector cosine search top-5 in project. Threshold:

- `similarity > 0.95` → **block** save (likely actual duplicate, suggest "view existing" link)
- `similarity 0.85 - 0.95` → **flag** with warning ("Near-duplicate of TC-RET-{N}, review before save")
- `similarity < 0.85` → save freely

**Performance:** ~50ms per candidate at PM1 scale (~50K test cases per project). HNSW index already built.

### 6.3 Token-Limit Safety (research-confirmed pattern)

For Composer ⓘ generations > 20 cases per call: **batch into chunks of 10**, parallel-call Groq, merge results. This avoids `gpt-oss-120b` 4K-output token cap. Implementation in `KbAnswerService` style — orchestrator + sub-call pattern.

## 10. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Groq RPD limit hit (1k/day across 8 users) | Medium | Hard fail (429s) | Auto-fallback to Gemini 2.5 Flash (1.5k RPD) + UI rate-limit warning |
| LLM returns invalid JSON despite schema | Low | Retry overhead | Retry once with stricter prompt; 2x fail = fallback to llama-4-scout; 3x = error to user |
| Generated cases too generic (low value) | Medium | UX complaint | RAG context (KB chunks) is the lever. Tune topK + prompt iteratively in Day-13/14. |
| Bulk import malicious CSV (formula injection) | Low | Data integrity | Sanitize on parse: reject rows with leading `=`, `+`, `-`, `@` per OWASP. Same as M2 KB sanitization. |
| Test case `key` (TC-RET-001) generation race | Low | Duplicate keys | Use Postgres SERIAL sequence (project-scoped) + retry on unique violation |
| pgvector HNSW slow as cases grow > 50K | Low (M3 won't hit this) | Slow dedup | M4 monitoring; if hit, add dedicated `test_case_embedding` table (TB-023) with own HNSW index |
