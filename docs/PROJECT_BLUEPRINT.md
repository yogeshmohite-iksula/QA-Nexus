## RISKY ASSUMPTIONS TO VALIDATE EARLY

1. **Technical — Render free dyno cold-start meets pilot UX bar.**
   PM1_ERD §10 acceptance gate 7 requires NFR-003 agent latency p95: A1 < 10s, A2 < 500ms, A4 < 15s. Render free spins down after ~15 min idle and takes ~10–15s to wake. UptimeRobot 5-min ping (MS0-T015) keeps it warm during 10 AM – 10 PM but the **first request of the day after the off-window** will cold-start. 8-user pilot can tolerate this; production cannot. Validate by: (a) confirming UptimeRobot is active before any pilot user signs in, (b) measuring p95 first-request latency in M5 pilot week, (c) documenting headroom for paid Render Starter ($7/mo) with explicit Yogesh approval if cold-starts impact the demo.

2. **Product — 8-user pilot validates B2B QA workflow.**
   PM1 ships entirely against one anchor project (Iksula Returns RET, Sprint 42 Day 9 of 14). The 8-user team is all internal Iksula. PM1_PRD §1 banks GA acceptance gate 10 on "6 of 8 pilot users complete create-requirement → generate-tests → run → triage → push-to-Jira flow without engineer intervention." Risk: this validates the workflow for ONE Jira instance, ONE project type (returns/refunds), and ONE company's QA culture. PM2-PM4 generalization to other Iksula projects (Commerce CART, Mobile App AUTH, Payments PAY, Internal Ops OPS) and external customers may surface workflow assumptions baked into PM1. Validate by: forcing a 1-day "second project" smoke test in M5 (e.g., Iksula Commerce) before declaring pilot success.

3. **Timeline — 18 weeks (or earlier) is achievable on a 3-FTE team with $0 infra.**
   M0 alone is ~270h across 3 FTE = ~30h/dev/wk for 2 weeks (sustainable). But M3 (Test Cases + A1 + A2) and M4 (Runs + Defects + Jira + A4 5-Layer RCA) are the AI-heavy milestones with eval gates: A1 ≥80% golden-set match, A2 <5% FP + ≥60% TP, A4 ≥70% top-2 RCA. **Eval failures in M3/M4 will burn weeks of prompt/retrieval iteration.** DeepEval on Colab Free is engineering-only and never blocks prod, but a failing golden set blocks the GA acceptance gate (PM1_ERD §10 gates 2/3/4). Validate by: building the M3 golden set in M0 (parallel work), running eval baselines in M2, accepting that any agent eval failing by 10pp by end of M4 triggers a milestone-slip discussion (NOT a paid-tier escape hatch).
