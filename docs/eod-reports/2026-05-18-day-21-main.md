## 6. Notes / decisions log

### Skill v2.2 — closed the structural-detection blind spot

The v2 heuristic (`STRUCTURAL_CLASS_HINTS` regex list) couldn't enumerate every per-frame BEM prefix (`sd-`, `sr-`, `def-`, `cl-`, `rs-`, etc.). v2.2's BEM heuristic generalizes — any `/^[a-z]+(-[a-z]+)+$/` class with structural content qualifies. F21 spec sectionLike count went from 13 (v2.1) → 46 (v2.2), +33 surfaced sections without false positives. The 5 Day-21 brief target patterns: 4 caught directly (sd-tabs, group-sep ×3, filter-strip, sd-section); the 5th (sr-layers at depth ~8) is left to the nested-count inverse probe as defense-in-depth.

### Day-21 cascade pattern — third repeat of CHANGELOG-conflict friction

Day-19 + Day-20 + Day-21 cascades all hit CHANGELOG conflicts on every rebased PR. Root cause: every PR adds an entry to `[Unreleased]`, so any two open PRs touch adjacent lines. Rebases compound the conflict. Day-21 was the worst — 4 PRs rebased + PR #177 rebased twice (once before, once after #176's version bump moved the `[Unreleased]` content to `[M4]`).

**M5 prevention candidates** (file as M5 hardening task):

1. **Changeset-per-PR strategy** — each PR adds `.changeset/<slug>.md` snippet; CHANGELOG assembled at release time via tool (e.g., `@changesets/cli`). Eliminates the `[Unreleased]` adjacency conflict class entirely.
2. **CHANGELOG-merge driver** — git custom merge driver that always concatenates additions instead of conflicting. Cheaper to implement; less ergonomic.
3. **Skip `[Unreleased]` for non-release-day work** — most Day-N EOD PRs don't need a CHANGELOG bump (already true per pre-push guard "no apps/\*\*/src/ touched"). Enforce by skipping the entry rather than authoring one.

Option 1 is the cleanest M5 hardening; bundle with the existing `@changelog-updater` subagent plan (P1.2 of the original audit doc).
