## 6. Notable learnings — four patterns codified today

### Pattern 1 — Halt-to-root-cause #1: R2 token scope (Step 1, AC012 unblock)

Symptom: pg_dump → R2 upload returned `AccessDenied`. Speculation bucket would have been: re-create token, recreate bucket, debug R2 config exhaustively. Halt-to-root-cause: walked `docs/deploy/restore-runbook.md`, recognized "second R2 bucket" mention, deduced existing token was scoped only to first bucket. Yogesh edited token's bucket scope (no key rotation); 6 min vs hours. **Lesson: read the runbook before guessing.**

### Pattern 2 — Halt-to-root-cause #2: ESLint config collision (Step 4, Path B → Path D switch)

Symptom: 40 `no-explicit-any` warnings in `apps/api/**/__tests__/*.ts`. Path B (initially authorized) would have added 80+ FIXME + eslint-disable annotations across 9 test files + filed a fake-debt followup. Halt-to-root-cause: noticed all 40 sites were in test files, read workspace-root ESLint config, found `no-explicit-any: 'off'` for test patterns at line 65, then found `apps/api/eslint.config.mjs` override at line 23-28 with `files: ['**/*.ts']` silently re-enabling `'warn'` for tests. Path D fix: 6-line config addition scoped to test patterns; 1 file vs 9; aligned with root intent; no debt. **Lesson: when symptom is "40 spurious warnings," check whether they're spurious before silencing them.**

### Pattern 3 — Halt-to-root-cause #3: CI flake → CI Postgres deficit (Steps B + C of close ceremony)

Initial speculation: bge-large embedding load timeout. Path α probe: re-run E2E; failed identically (eliminated flake). Path β: changed `EMBEDDING_MODEL_ID` to bge-small via PR #23; failed identically (falsified bge-large hypothesis — main `68b3ac0` had passed earlier with bge-large). Halt-to-root-cause: instrumented with Path δ (`if: always()` log dump + system snapshot in PR #23); next failed run revealed `PrismaClientInitializationError: Can't reach database server at localhost:5432`. Real cause: structural CI Postgres deficit (no `services: postgres:` block); fixed in PR #24. **Lesson: instrument before iterating, name the falsified hypothesis explicitly so it stays falsified.**

### Pattern 4 — Refined pre-merge gate rule (PR #24 evidence, Step C)

Strict "no continue-on-error masking" pre-merge rule was over-conservative — tripped on PR #24 even though PR #24 was IMPROVING signal (7 of 11 onboarding tests went F→P thanks to the Postgres fix). Halt-to-escalate moment: surfaced to Yogesh ("strict reading blocks; this PR is making things better"). Yogesh refined the rule: **compare failure-set on PR vs failure-set on main; only NEW failures introduced by the PR trigger the halt. Pre-existing inherited failures get filed as followups, not blockers.** Applied immediately to PR #22 in Step D — PR #22's failures matched main's `:38`/`:44` set exactly → merged cleanly. **Lesson: halt-to-escalate even on rules I wrote myself; rules can be wrong; evidence wins.**

**Shared discipline across all four:** SURFACE the discrepancy + ASK before bypassing. Prefer evidence over speculation. Instrument before iterating. **Halt-to-escalate even on your own rules.** This discipline is now expected behavior on QA Nexus per project culture.
