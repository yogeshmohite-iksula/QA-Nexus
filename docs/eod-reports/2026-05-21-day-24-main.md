## 7. Lessons learned (Day-24 retro candidates)

### 7.1 Bundle workflow is FRAME-SPECIFIC quality (the key ADR-022 input)

F22 REJECTED + F25 PASSED across 2 consecutive days proves bundle workflow output varies frame-by-frame. The pre-Step-3 sanity check (count v2 HTML sections vs bundle spec.json; reject if > 30% divergence) is what makes the workflow SAFE to attempt. Without it, F22's late-Step-6 surprise would have repeated on every frame with subtle structural divergence. ADR-022 ratification framing: **adopt the sanity check as a binding contract; bundle is an opt-in optimization for frames where the check passes; Rule 15 + skill v2.2 fallback for frames where it fails.**

### 7.2 Pre-stage scaffold pattern (3× validated this week — promote to memory file)

Day-22 walkthrough → Day-23 ratification 10 min. Day-23 EOD prep → Day-23 commit 5 min. Day-24 close plan + close report skeleton → Day-25 EOD mechanical. ROI 2-3× per use. Promote to `feedback_pre_stage_scaffold_pattern.md` at M5 close — establishes the pattern as standing Day-N+1 buffer prep protocol. Cross-references: `feedback_session_resume_state_verification.md` + `feedback_branch_lineage_drama.md` siblings under "single-human × N-window coordination friction" umbrella.

### 7.3 Branch-lineage drama recurrence — Day-24 zero incidents

Day-22 ~30 min recovery → Day-23 ~3 min → Day-24 ~0 min. **Muscle memory + cat-heredoc workaround now consistently applied.** Day-24 P0a chained-base verification (Yogesh + BE+1 confirmed flat-base for #189/#194/#195) is the OFFENSIVE version of the workaround — verify base ref BEFORE cascade rather than recover AFTER. Pre-tool-use hook proposal `git rev-parse HEAD == <expected SHA>` would have caught all Day-22-23 incidents at 0 cost; recommend Day-26+ as the first M6 hygiene task.

### 7.4 Day-25 cascade simplification from Branch B (~15-30 min) to Branch B-only (~15 min)

The Day-22 chained-base squash gotcha taught the team to anticipate auto-close risk. Day-24 PM Yogesh+BE+1 verified flat-base BEFORE the cascade hit, simplifying the recovery budget from "15-30 min depending on chain depth" to "~15 min sequential rebase". **The recovery memory's value is also as a prevention checklist** — re-read it BEFORE the next cascade, not AFTER.

### 7.5 F19 [PARTIAL] checkpoint shape is the cleanest mid-skill checkpoint pattern

Steps 1-2 committed with `canned-data.v2.ts` + `F19.spec.json` on disk → Day-25 AM resumes at Step 4 zero-cost. **Codify the [PARTIAL] = Steps 1-2 committed shape** in `feedback_skill_v2.2_first_use.md` Day-26+ as the recommended mid-skill checkpoint. Avoids both "all-or-nothing" pressure AND "phantom carryover that requires re-extraction".

### 7.6 Memory-file conflict resolution: separate-file pattern for parallel-agent additions

When two agents edit the same memory file with non-overlapping findings, the conflict resolves cheapest via **separate file names** — not in-place merge.
