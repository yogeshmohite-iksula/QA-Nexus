## 6. Lessons learned (Day-28)

### 6.1 LLM-assist corpus labeling provenance

Codex-generated labels with human spot-check (6 sub-0.70 confidence cases) shipped as the AC042 ground truth. Per ADR-022 §5.9 reserve-allowance precedent, the trade-off is documented. M6+ corpus governance should require an explicit **LLM-assist-vs-pure-human flag on every eval run** — current eval-gate semantics now measure LLM-Sherlock vs LLM-Codex agreement, NOT vs pure-human ground truth. Track for an M6 corpus-governance ADR. (Mirrors close-report §6.9.)

### 6.2 Schema-bridge smoke harness as pre-eval sanity gate

AC042 binding eval Day-27 burned ~60 min full-corpus run on a Zod-validation bug that a 1-defect smoke (4 Groq calls, ~15s) would have caught. New `scripts/ac042-smoke.mjs` (or `AC042_LIMIT` env on the harness) now exists; future eval-gate runs should fire smoke BEFORE binding eval. M6 ADR-022 amendment candidate. (Mirrors close-report §6.8.)

### 6.3 Branch-name collision via worktree cross-pollination

MAIN agent's Day-27 EOD commit `bb4fe51` landed on `feat/be-ac042-corpus-labels-codex` (BE+1's intended branch) before BE+1 pushed, forcing BE+1 to pivot to a `-be1` suffix. Worktree cross-pollination at the harness layer (`git push` simple-mode targeting an unintended branch). Phase E cleanup: `gh api -X DELETE repos/yogeshmohite-iksula/QA-Nexus/git/refs/heads/feat/be-ac042-corpus-labels-codex`. **Hook mitigation candidate for M6:** pre-branch-create check that warns if branch name already exists on a sibling worktree's local refs, OR EOD push scripts use `git push origin HEAD:refs/heads/<expected-branch>` to lock target explicitly.

### 6.4 MAIN-owned inline conflict resolution beyond the runbook (Day-28 cascade)

Risk-4 memory-file resolution (#193) and the xlsx binary conflict (#207) both required MAIN inline conflict resolution beyond the runbook. **Pattern:** when MAIN owns the conflict resolution, document the union/canonical decision in the commit message + memory file (for retro evidence). **Adopted:** docs/scripts that produce binary outputs (xlsx work-log) should regenerate-from-source in Phase E rather than merge binary diffs — binary 3-way merge is impossible, so each PR touching the xlsx conflicts on the second merge. Day-29 Phase E does one authoritative `update-work-log.py` regeneration instead.
