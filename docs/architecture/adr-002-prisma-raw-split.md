# ADR-002: Split Prisma migrations between `prisma/migrations/` and `prisma/raw/`

- **Status:** Accepted
- **Date:** 2026-04-28
- **Deciders:** Yogesh Mohite (Admin), BE chat
- **Related:** PR #4 `a6644c1` (where the issue surfaced) · `docs/followups.md` entry (a) · PM1_ERD §3 (table inventory) · PM1_ERD §3.13 (audit log triggers)
- **Supersedes:** none
- **Superseded by:** none

---

## Context

During MS0-T020 (Prisma schema for TB-001..TB-021), the BE chat hand-wrote infrastructure SQL — RLS policies, HNSW indexes, the `pgvector` extension, and audit-log triggers — and tried to land it as a regular Prisma migration at `apps/api/prisma/migrations/0_init_rls_hnsw/migration.sql`.

This failed in two ways:

1. **Shadow-DB validation failure.** Prisma's `prisma migrate dev` validates migrations by replaying them in lex order against a freshly-created shadow database. The hand-written file referenced tables (e.g. `audit_log`, `user`) that hadn't been created yet by Prisma's auto-generated init migration — Prisma had not yet emitted that init migration, because we were trying to land both at once.
2. **Re-application semantics.** Some hand-written DDL is idempotent (`CREATE EXTENSION IF NOT EXISTS pgvector`), some is not (`ALTER TABLE ... ENABLE ROW LEVEL SECURITY` will error if the policy already exists). Prisma's migration runner expects each file to be applied exactly once and tracks state in `_prisma_migrations`. Mixing one-shot and idempotent SQL inside Prisma's tracking model is fragile.

The minimal fix during PR #4 was to **relocate the file** from `apps/api/prisma/migrations/0_init_rls_hnsw/migration.sql` to `apps/api/prisma/raw/init_rls_hnsw.sql` and apply it manually via `prisma db execute --file prisma/raw/init_rls_hnsw.sql --schema prisma/schema.prisma`. That fix shipped at commit `a6644c1`. This ADR formalizes the convention going forward, because **MS0-T021 (BetterAuth)** is about to add session/account/verification tables that also need RLS policies — we need a clear rule before that lands.

## Decision

Adopt a **two-directory split** for all SQL touching the database:

| Path                                 | Owner                                                | Applied by                                  | Tracked in             |
| ------------------------------------ | ---------------------------------------------------- | ------------------------------------------- | ---------------------- |
| `apps/api/prisma/migrations/<ts>_*/` | Prisma (auto-generated by `prisma migrate dev`)      | `prisma migrate deploy` in CI               | `_prisma_migrations`   |
| `apps/api/prisma/raw/*.sql`          | Hand-written infra (RLS, HNSW, extensions, triggers) | `prisma db execute` via `pnpm db:apply-raw` | git history (per file) |

**Rules for `prisma/raw/*.sql`:**

1. **Idempotent by default.** Use `IF NOT EXISTS`, `CREATE OR REPLACE`, `DROP POLICY IF EXISTS … ; CREATE POLICY …`, etc. The file MUST be safe to re-apply against a database that already has it applied — this is the contract that lets us trust git history alone, with no `_raw_migrations` tracking table.
2. **One file per logical concern.** `init_rls_hnsw.sql` for the v0 platform infra; future BetterAuth RLS goes into `betterauth_rls.sql`; future audit-log trigger updates go into `audit_log_triggers_v2.sql`. Don't append to existing files except for trivial fixes — create a new file so re-apply order is clear.
3. **Reference Prisma-managed tables only AFTER they exist.** A `prisma/raw/*.sql` file may freely reference tables that Prisma created (e.g. `ALTER TABLE "user" ENABLE ROW LEVEL SECURITY`). The execution order is: `prisma migrate deploy` first, `pnpm db:apply-raw` second. Never the other way around.
4. **Apply in CI explicitly, not implicitly.** A future CI workflow will run `pnpm --filter api prisma:migrate:deploy && pnpm db:apply-raw` against the staging database. Until that workflow lands, applying to production is a **manual step** with human approval (and an entry in `docs/eod-reports/` documenting it). No silent automation of RLS changes.

## Consequences

**Positive:**

- Prisma migrations stay clean, lex-ordered, and shadow-DB-validatable.
- Hand-written infra is visible in code review (it's a `.sql` file, not buried in a Prisma migration directory).
- Re-applying `prisma/raw/*.sql` files is safe — engineers can run `pnpm db:apply-raw` on a fresh dev DB or against staging without fear of state corruption.
- Pairs naturally with `pnpm db:apply-raw` (added in this same change-set) as the executable shorthand for the manual application step.

**Negative / accepted trade-offs:**

- No automatic tracking of which raw files have been applied to which environment. Mitigated by (a) idempotency contract above, and (b) git history showing when each file landed on `main`.
- Engineers must remember to run `pnpm db:apply-raw` after `prisma migrate deploy` when raw files have changed. Mitigated by adding a checklist item to the deploy runbook (`docs/deploy/`).
- Slightly more setup ceremony than a "single source of truth" approach. Accepted because the alternatives (below) all lose more than they gain.

**Neutral:**

- BetterAuth's session/account/verification tables (MS0-T021) will follow this exact pattern: Prisma owns the tables, `prisma/raw/betterauth_rls.sql` owns the row-level security on them.

## Alternatives considered

### A. Put everything in `prisma/migrations/` and disable shadow-DB validation

Set `previewFeatures = ["preserveScalar"]` and run `prisma migrate dev --skip-shadow-db`. **Rejected** because (a) shadow-DB validation is the only thing preventing accidental destructive migrations, and (b) Prisma still tracks each migration as one-shot, which breaks our "RLS files are idempotent" contract.

### B. Use a separate migration tool (e.g., `dbmate`, `goose`, `sqitch`) for raw SQL

**Rejected** because it adds a third tool to the deploy pipeline (Prisma + dbmate + Husky), more env vars, more learning surface for the team. We're 8 users in a $0/month build — minimum tool count wins.

### C. Track raw-file application in a `_raw_migrations` table with file hash + applied_at

**Considered, deferred.** This is the "right" engineering answer at scale but adds ~50 LoC of bootstrap (create the table, hash files at apply time, skip-if-already-applied logic) for ~zero benefit at 8 users on a single Neon DB. Revisit if (a) we add a second environment beyond `dev` + `prod`, or (b) raw-file count exceeds 5.

### D. Inline RLS into the Prisma schema via a forthcoming `@@policy` directive

**Rejected** because the directive doesn't exist in Prisma 5 (locked-major). Even if Prisma 6 adds it, we wouldn't take the major bump just for this.

## Open questions resolved by this ADR

The three questions raised in `docs/followups.md` entry (a):

1. **How is `prisma/raw/` applied in CI?** Manually for now (with human approval gate); add `pnpm --filter api prisma:migrate:deploy && pnpm db:apply-raw` to the deploy workflow when one exists. Documented as a deploy-runbook checklist item, not yet automated.
2. **How is "applied" state tracked?** Trust git history per file. Idempotency is the contract that makes this safe (Rule 1 above).
3. **Re-apply semantics?** Idempotent by default (Rule 1). The current `init_rls_hnsw.sql` will be audited against this rule in a follow-up commit (it currently mixes idempotent and one-shot — needs cleanup).
