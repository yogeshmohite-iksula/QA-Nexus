## Step 4 — First deploy (and the gotchas)

Click **Create Web Service**. Render will:

1. Clone the repo at `main` HEAD.
2. Run the build command (`pnpm install && pnpm --filter @qa-nexus/shared build && pnpm --filter @qa-nexus/api build`). Expect ~3-4 min on first deploy (cold pnpm cache).
3. Run the start command. NestJS bootstraps + binds to `$PORT`.
4. **Important: the embedding model (`@xenova/transformers` + `Xenova/bge-large-en-v1.5`) loads on bootstrap.** First boot downloads ~660 MB of ONNX assets from Hugging Face → takes ~38s. Subsequent boots use the local FS cache (~1.7s). Render's free dyno has 512 MB RAM, but the model fits because it's WASM (~280 MB resident at warm).

### First-deploy gotchas

**Gotcha 1: Prisma migration not auto-run.** Render's build command does NOT run `prisma migrate deploy`. The schema must already exist on Neon. **Fix:** before first deploy, run from your local:

```bash
DATABASE_URL=<neon-pooled-url> DIRECT_URL=<neon-direct-url> pnpm db:migrate:deploy
DATABASE_URL=<neon-pooled-url> DIRECT_URL=<neon-direct-url> pnpm db:apply-raw
```

Both schemas (Prisma migrations + raw RLS/HNSW) need to land before the API can serve auth requests.

**Gotcha 2: BETTER_AUTH_URL chicken-and-egg.** Render assigns the URL AFTER first deploy succeeds. So `BETTER_AUTH_URL` is unset on first deploy → BetterAuth logs a warning + falls back to `http://localhost:3001`. Magic-link emails will reference localhost. **Fix:** after first deploy succeeds (any HTTP response from /health), copy the assigned URL (e.g. `https://qa-nexus-api.onrender.com`) into the env-var panel, save, click "Manual Deploy → Deploy latest commit". Second deploy uses the right URL.

**Gotcha 3: Embedding model OOM on free dyno.** 512 MB RAM is tight when (a) NestJS + Prisma + (b) the bge-large model are both warm. If you see `WORKER OOM` in logs:

- Switch `EMBEDDING_MODEL_ID` to a smaller model temporarily: `Xenova/bge-small-en-v1.5` (384-dim, ~100 MB resident). **Note:** this changes vector dim — schema migration required (`vector(1024)` → `vector(384)`). Don't do this lightly. Better path: upgrade to a paid Render dyno temporarily during pilot scale-up.
- OR: lazy-load the model on first /embedding request instead of eager-load at bootstrap. Code change: `apps/api/src/embedding/embedding.service.ts` — defer `pipeline()` call to first `embed()` call. Costs ~38s latency on the first request post-deploy but keeps boot under 5s.

**Gotcha 4: Cold-start latency.** Render free dyno sleeps after 15 min of no requests. Cold-start = ~10-15s (including model warm-up if cache survives, ~40s if cache evicted). UptimeRobot pings /health every 5 min during operating window (10 AM – 10 PM IST) to keep the dyno warm. See `uptimerobot-runbook.md`.

## Step 5 — Post-deploy verification

Once Render reports "Live" status, run these from your local terminal (replace URL with the actual assigned one):

```bash
# 1. Basic reachability
curl -s -o /dev/null -w "HTTP %{http_code}\n" https://qa-nexus-api.onrender.com/health
# Expected: HTTP 200

# 2. Full health readout
curl -s https://qa-nexus-api.onrender.com/health | jq
# Expected fields:
#   status: "ok"
#   db: { up: true, latency_ms: <number> }
#   embedding: { warm: true, model: "Xenova/bge-large-en-v1.5", dim: 1024 }
#   r2: { status: "deferred", reason: "missing credentials" }    ← until T013
#   llm: { status: "deferred", reason: "missing API keys" }      ← until T023
#   quota: { db_size_mb: 9.4, db_size_pct: 1.83 }

# 3. Public auth endpoints reachable
curl -s -o /dev/null -w "sign-up:  HTTP %{http_code}\n" https://qa-nexus-api.onrender.com/auth/sign-up
curl -s -o /dev/null -w "sign-in:  HTTP %{http_code}\n" https://qa-nexus-api.onrender.com/auth/sign-in
# Expected: both 400 (missing body) — proves the routes exist and Nest is up

# 4. RBAC blocks anon to protected endpoint
curl -s -o /dev/null -w "rbac-demo:  HTTP %{http_code}\n" https://qa-nexus-api.onrender.com/rbac-demo/admin-only
# Expected: 401 (not authenticated)
```

If all four pass, **paste the URL back into this chat** and MAIN will:

2. Update `docs/STATUS.md` Last Deploy + Health Check rows.
3. Update `BETTER_AUTH_URL` in Render env vars (you can paste it once you've shared the URL).
4. Trigger a re-deploy so BetterAuth picks up the URL.

## Step 6 — Rollback procedure

Render keeps the last 5 successful deploys. To roll back:

1. Render dashboard → service → **Events** tab.
2. Find the last good deploy (look for green "Live" badge).
3. Click the deploy → **Rollback to this deploy**.
4. Render redeploys the previous Docker image. ~30s downtime.
5. **Important:** rollback does NOT undo database migrations. If a migration broke prod, you also need to run `prisma migrate resolve` from your local — see `docs/architecture/adr-002-prisma-raw-split.md` § "Re-apply semantics".

## Step 7 — Common errors + fixes

| Symptom                                                    | Likely cause                           | Fix                                                                                      |
| ---------------------------------------------------------- | -------------------------------------- | ---------------------------------------------------------------------------------------- |
| Build fails: `Module not found: @qa-nexus/shared`          | Build command didn't pre-build shared  | Verify build command exactly matches Step 2 table (chains shared build first)            |
| Boot fails: `Cannot find module '../shared/dist'`          | Same as above                          | "                                                                                        |
| Boot fails: `MODULE_NOT_FOUND: better-auth/node`           | esModuleInterop config issue           | Verify `apps/api/tsconfig.json` has `"esModuleInterop": true` (already set Day 2)        |
| Logs show `Falling back to localhost:3001` from BetterAuth | `BETTER_AUTH_URL` unset (first deploy) | Paste the Render URL into env vars + redeploy                                            |
| Logs show `WORKER OOM` after ~40s                          | Embedding model OOM on 512 MB          | See Gotcha 3 above                                                                       |
| `/health` returns 503 + db.up=false                        | Neon connection limit (100) hit        | Check Neon dashboard "Connections" — stale connections from local dev? Restart local API |
| Dyno restarts every ~10s                                   | Build healthcheck failing              | Render's deploy healthcheck = `GET /health`. Make sure it returns 200 on first call.     |
